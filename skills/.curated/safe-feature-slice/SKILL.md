# Safe Feature Slice

## Core Principle

Optimize for preserving the feature invariant, avoiding unsafe states, keeping business rules in the correct layer, making the code testable, preventing integration drift, and not inventing hidden product decisions.

Do not redesign the feature unless explicitly asked. If something is ambiguous, write down the ambiguity and recommend a safe default instead of silently choosing a risky path.

## Initial Read

Read the relevant files before coding or reviewing. Look for:

- Feature brief or decision notes
- Existing tests and fixtures
- Previous intended behaviours that should remain true after this slice
- Domain logic, shared types, and contracts
- Database schema, migrations, constraints, RLS, and policies
- API routes, server actions, webhooks, jobs, and queues
- Auth checks, ownership checks, permissions, and audit logs
- UI flows, error handling, retry/idempotency behavior, and stale-version handling

If no formal feature brief exists, infer a short working version before coding or reviewing:

```text
Feature:
[what the user should be able to do]

Primary actor:
[who is allowed to do it]

Core invariant:
[this must always remain true]

Previous intended behaviours:
[existing goals or guarantees that should remain true]

Unsafe outcomes:
[what must never happen]

Assumptions:
[what I am assuming because it was not specified]
```

## Audit-Driven Slice Inputs

When the slice scope comes from an external audit — a manual walkthrough, persona-review report, security scan, accessibility audit, gap report, or any artifact authored before recent code changes — do not treat its findings as gospel. Audits go stale the moment the next commit lands.

For every finding pulled into the slice scope:

1. Reproduce the finding against the current code before adding it to the slice. Hit the route, run the failing test, render the screen, or grep the symbol the finding names.
2. If the finding cannot be reproduced, mark it as `verified-resolved` and drop it from scope. Note the verification method (e.g., "curled the route, no error string in response body").
3. If the finding partially reproduces but the symptom or root cause differs, treat it as a new finding and re-scope.
4. If the audit data was generated by a script or instrumentation, sanity-check the script. False-alarm findings are common when the probe has its own bugs.

This step costs minutes and avoids the larger cost of fixing things that aren't broken. It is especially important when the audit captured screenshots — a screenshot is a snapshot in time and can show layout artifacts (fixed elements rendered mid-page in `fullPage` captures, dev-only indicators, scroll-reveal animation states) that real users never see.

## Multi-Slice Gate

This skill is the front door for both planning and executing safe feature work. One narrow slice remains the atomic implementation unit, but broad work should be planned inside this workflow before execution instead of bouncing out and back.

When a plan contains multiple required slices, execute them as a slice loop by default: finish one slice, record evidence, then continue to the next eligible slice until all required slices are done, explicitly skipped, or blocked.

Before coding or reviewing, classify the request as one of:

- `single-slice`: one actor/trigger, one main action, one invariant, one acceptance signal, small expected file surface.
- `planned-slice`: the request points at an existing `plans/<feature-slug>/slice-plan.md` slice or the repo already has a relevant slice plan with a clear next eligible slice.
- `needs-plan`: broad feature, multi-step fix, roadmap, hardening sweep, or work spanning several independently verifiable steps.

### Auto-Routing Rules

Proceed directly in this skill when the request is `single-slice`.

Use the existing plan when the request is `planned-slice`:

- Find the relevant `plans/<feature-slug>/slice-plan.md` from the user request, current branch/docs, or nearby plan files.
- Select the explicitly requested slice if one is named.
- Otherwise select the first `pending` slice whose dependencies are `done` or not required.
- If a `blocked` slice appears unblocked from new evidence or a user decision, update the plan blocker first, then select it.
- Move only the selected slice into the active working context.
- After that slice reaches `PASS`/`done`, update the plan and continue to the next eligible required slice unless the user explicitly said to stop after the current slice or a stop condition applies.
- Do not merge later slices into the current slice just because they are listed.
- Update the plan progress before starting and after verification when edits are made.

Create or update `plans/<feature-slug>/slice-plan.md` inside this workflow when the request is `needs-plan`. Use the thin-slice planning discipline:

- start with a working brief: feature/fix, actors, invariant, previous behaviours, unsafe outcomes, evidence, assumptions, out of scope
- classify plan and slice risk tiers
- build a dependency graph before sequencing
- add a progress table with `pending`, `in_progress`, `blocked`, `done`, and `skipped`
- define each slice with actor/trigger, action, invariant, intentional behaviour changes, previous behaviours preserved, unsafe outcomes, dependencies, expected files, write boundaries, tests, runtime verification, migration/external notes, acceptance criteria, exit evidence, and parallelization notes
- mark destructive or live-data mutations as blocked until explicitly approved

Use the standalone `thin-slice-plan` skill only when the user explicitly wants planning-only output, asks to review a plan without implementation, or the active workflow should stop after creating a plan.

After the plan exists, continue in this skill for one selected slice at a time inside a continuing execution loop unless the user asked for planning only. Do not silently expand the active slice to absorb adjacent plan items. If implementation reveals new required work, update the slice plan rather than hiding it inside the current slice, then continue through the revised eligible slice list unless blocked or told otherwise.

If classification is ambiguous, prefer the safest reversible path:

- If the work could affect money, permissions, ownership, destructive writes, state transitions, webhooks, migrations, or customer-visible records, treat it as `needs-plan` unless repo evidence proves it is one slice.
- If the user named an exact failing test, file, route, or behavior and the blast radius is narrow, treat it as `single-slice`.
- If a plan artifact exists and has a clear next slice, treat it as `planned-slice`.

### Routing Examples

| Request shape | Classification | Action |
| --- | --- | --- |
| "Fix this failing auth test in `tests/auth/foo.test.ts`" | `single-slice` | Execute one tested slice. |
| "Add quote approvals with staff submit, admin approve, locked approved quotes, and audit logs" | `needs-plan` | Create/update `plans/<slug>/slice-plan.md`, then execute the first eligible slice if implementation was requested. |
| "Continue the quote approval plan" with `plans/quote-approval/slice-plan.md` present | `planned-slice` | Pick the next eligible slice from the plan. |
| "Review this PR for payment state safety" | `single-slice` if PR is narrow, otherwise `needs-plan` | Classify after the initial read. |
| "Implement all remaining slices" | `planned-slice` | Execute the next eligible slice, then keep going until the required slice set is done, skipped, or blocked. |

## Blocker Triage Gate

When a blocker is discovered during a plan, slice, verification node, browser proof, test run, deploy watch, or review, stop it from becoming a vague note. Triage it before expanding the active slice.

Classify the blocker as one or more of:

- product decision
- repo bug
- test or verification failure
- env or data drift
- migration, RLS, schema, or permission drift
- external service, connector, access, billing, or quota issue
- unsafe live-data or destructive-change risk

Record the blocker with enough evidence that another workflow can act without rediscovering it:

- affected plan node, slice, route, or feature
- exact symptom
- repro command, URL, or user action
- expected behavior
- actual behavior
- concise log excerpt or error text
- suspected files, migrations, services, or external system
- why it blocks the graph or slice
- explicit unblock condition

Route the blocker deliberately:

- If it is small, deterministic, and inside the active slice, fix it inside the slice and keep the evidence with that slice.
- If it blocks future work but is outside the active slice, add or update the blocker row and do not silently expand scope.
- If diagnosis is unclear, cross-cutting, or likely to need prioritization, hand it to `issue-fix-strategy` with the handoff packet below.
- If it needs product, business, route, permission, or lifecycle input, mark it `decision-needed` and ask the smallest useful question.
- If it needs live data, auth, role, billing, production config, destructive migration, or external account mutation, pause for explicit approval.

Use this `issue-fix-strategy` handoff packet when routing a blocker:

```text
Blocker:
[short title]

Affected node or slice:
[plan node, route, or slice id]

Symptom:
[what failed]

Repro:
[command, URL, or browser action]

Expected:
[expected behavior]

Actual:
[actual behavior and concise error]

Suspected cause:
[files, migrations, services, env, provider, or unknown]

Why it blocks:
[which future work cannot proceed safely]

Unblock condition:
[what evidence means this is resolved]
```

## Interview Gate

After the initial read, decide whether the slice is specified enough to proceed safely.

Proceed without interviewing when repo evidence or the user brief identifies the actor, allowed action, core invariant, unsafe outcomes, integration boundaries, and acceptance signal, or when remaining ambiguity can be handled by a safe reversible default.

Interview the user before coding or final review when a missing answer could change auth, ownership, money, state transitions, destructive behavior, data model, external contracts, customer-visible behavior, migration scope, or deploy risk.

Keep the interview short:

- Ask 1-5 numbered questions.
- Phrase each question as the decision needed, with the safest default if the user wants to proceed.
- Do not ask questions that can be answered from repo evidence.
- If the user asked not to be interrupted, record assumptions and continue only with reversible or low-risk work; mark `BLOCKED` for unsafe decisions.

Classify the risk tier before proceeding:

- Tier 1: money, permissions, data ownership, state transitions, destructive writes, webhooks, accepted/customer-facing records.
- Tier 2: external adapters, background jobs, queues, sync logic, admin workflows.
- Tier 3: UI display, copy, layout, simple filters, low-risk polish.

Tier 1 requires stricter review, stronger tests, and explicit safety checks. Tier 3 should stay proportionate.

## Subagent Orchestration Gate

For non-trivial slices, start with a dependency graph: what is blocking, what can be checked independently, and what must stay single-threaded.

Use subagents when they can reduce risk or wall-clock time without mutating the same state. Keep the critical path local and delegate independent specialist work such as:

- Repo exploration for a narrow surface or contract.
- Auth, ownership, RLS, migration, or state-transition audit.
- Test gap review or focused verification planning.
- UI/runtime verification when another agent can safely operate a browser or Computer Use surface.
- Third-party API/framework check against Context7 or current official docs when current behavior matters.

For planned slice execution, each implementation slice should be done by an independent worker agent when subagents are available and safe. Use a fresh worker per slice so context, evidence, and ownership stay clean. If dependencies force sequential work, start the next independent worker only after the previous slice clears its gates. If multiple slices are truly parallel-safe, give each worker disjoint write ownership.

Reasoning budget:

- Main coordinator: use extra-high reasoning (`xhigh`) when the runtime lets the main session choose reasoning effort.
- Subagent workers: spawn with model `gpt-5.5` and medium reasoning effort by default; choose the fastest available medium profile when the runtime exposes speed/profile controls.
- Raise a worker above medium only when the assigned subtask is unusually ambiguous, high-risk, or architecture-heavy; state why in the brief.
- If the runtime does not expose model, reasoning, or speed controls, still follow the coordinator/worker split and do not block on the setting.

When delegating, give each subagent a specialist brief with role, exact goal, repo/path ownership, read/write boundaries, context, expected evidence, and report format. For code-writing subagents, assign disjoint files or modules and state that they are not alone in the codebase, must not revert others' work, and must adapt to nearby changes.

Do not parallelize shared-state operations: git checkout/merge/push, package lockfile edits, migrations against the same database, test runs sharing one mutable database, deploys, or multiple agents writing the same files. Do not delegate unresolved product decisions; use the Interview Gate instead.

Subagents must report concise evidence: files read or changed, commands run, tests/checks performed, findings, unresolved risks, and handoff notes. The main agent owns integration, final judgment, and the final `PASS`/`BLOCKED`/`FAIL` status.

## Build Mode

Use build mode when the user asks to implement or harden one slice. Keep the slice narrow and state this before edits:

```text
Slice:
[what this slice does]

Intentional behaviour changes:
[what the request is meant to change]

Previous intended behaviours preserved:
[existing goals or guarantees that should still hold]

Allowed scope:
[files/folders expected to change]

Invariant protected:
[safety rule this slice must preserve]

Dependencies:
[what this assumes already exists]

Integration risk:
[how this could break another part]
```

Then work in this order:

1. Find or write the failing test first.
2. Confirm the test fails for the right reason.
3. Implement the smallest correct change.
4. Keep business rules out of UI components and thin route handlers.
5. Use existing shared types/contracts; do not redefine them locally.
6. Add edge-case tests for unsafe outcomes.
7. Add or run regression checks for previous intended behaviours that should remain true.
8. Run the relevant automated checks and required browser/computer-use verification.
9. Report what changed, what intentionally changed, what prior behaviours were preserved, and what remains risky.

Prefer boring, explicit, testable code. Use domain/service layers for critical rules. Use database constraints, RLS, transactions, idempotency keys, audit events, and stale-version checks where the risk tier calls for them.

## Review Mode

Use review mode when the user asks for a review, audit, safety pass, or already-built feature assessment. Do not only judge whether the code looks fine. Apply the decision gate, then review against the safety questions.

### Review Decision Gate

During review, do not interview for issues that repo evidence can prove or where the intended behavior can be safely inferred from existing code, tests, fixtures, docs, schema, product copy, or adjacent workflows. Keep moving and cite the evidence.

Interview only when a missing product, business, or operational answer could change:

- Whether the behavior is a bug or intended.
- Who should be allowed to perform the action.
- The correct ownership, money, status, destructive-write, or customer-visible rule.
- Whether a migration, backfill, deploy, or live verification step is safe.
- Whether the final status should be `PASS`, `PASS WITH RISKS`, `BLOCKED`, or `FAIL`.

Classify candidate issues as:

- Confirmed finding: repo/runtime evidence proves the unsafe path.
- Missing evidence: more local repo or runtime investigation is needed before asking the user.
- Decision needed: the user must confirm a product, business, or operational rule.

For decision-needed issues, ask 1-3 numbered questions after the evidence pass. Include the safest default and the consequence of using it. Do not use questions as a substitute for reading the code, and do not pause for minor ambiguity that can be handled by a safe, reversible default.

Review against these questions:

- What is the core invariant?
- Which previous intended behaviours should remain true?
- Which behaviour changes are intentional because of this request?
- Could the implementation satisfy the new request while drifting the older goal?
- Can this feature enter an unsafe state?
- Can the wrong user access or mutate the wrong data?
- Can the frontend trick the backend?
- Can state change in more than one place?
- Can money, ownership, permissions, or status change without audit?
- Can duplicate requests, retries, or webhooks cause duplicate effects?
- Can stale data or old versions be accepted?
- Are errors swallowed instead of handled?
- Are database constraints and RLS doing enough?
- Do tests prove dangerous cases or only the happy path?
- Does this need browser or Computer Use verification to prove the real workflow?
- Would this still work if two requests happen at the same time?
- Does this slice break the feature spine or shared contract?

Then run a separate structural maintainability pass, especially when the diff touches shared routes/services, grows large files, adds tooling, or threads new state through an existing workflow:

- Is there a simpler reframing that would delete branches, helpers, modes, or repeated plumbing while preserving behavior?
- Did feature-specific logic leak into a shared route, proxy, service, renderer, or helper that now knows too much?
- Did the change duplicate a canonical loader, renderer, policy, or truth table instead of testing or reusing the existing one?
- Did it add special-case conditionals to an already busy flow where a dedicated model, policy, or boundary would be clearer?
- Did it make an already-large file absorb more concerns, or push a file past roughly 1000 lines without a strong structural reason?
- Did it introduce loose object shapes, optionality, casts, or fallback parsing where an explicit typed boundary would be safer?
- Do new scripts or test tools write artifacts into the repo root or unignored paths instead of a contained, ignored output directory?

Treat evidence-backed structural regressions as review findings, not cosmetic nits. A correctness-only `PASS` is not enough when the implementation creates likely future regressions, hides duplicated truth, or normalizes artifact sprawl. Use `PASS WITH RISKS` for contained debt with a clear follow-up; use `FAIL` when the structure already caused or strongly enables a user-visible, data, auth, deploy, or workflow regression.

For code reviews, lead with concrete findings and file/line references. Classify confirmed unsafe paths separately from hypotheses and missing-test risks.

## Hard Rules

- Do not redefine shared types locally.
- Do not invent new states casually.
- Do not bypass the domain layer.
- Do not put critical business rules only in UI components.
- Do not trust request body values for ownership, price, amount, role, status, or permissions.
- Do not use `as any`, `@ts-ignore`, or disabled lint rules to force progress.
- Do not delete or weaken tests.
- Do not mock the dangerous part when it should be tested for real.
- Do not silently catch errors and continue.
- Do not add fallback behavior for unsafe cases.
- Do not use service-role/admin access unless explicitly justified.
- Do not change contracts without calling it out.
- Do not merge unrelated cleanup into the slice.
- Do not mark `PASS` when browser/computer-use verification was required but not performed.
- Do not start optional follow-up work outside the active slice plan just because it is recommended. Required planned slices are not optional follow-up work; continue through them unless blocked or told to stop.
- Do not fix audit-driven findings without first reproducing them against the current code. Audits go stale.
- Do not declare a fix safe based on tsc/CI passing against stale or missing generated files. Reproduce the failure mode against fresh generated state at least once before claiming green.

## Safety Checks

For features touching money, permissions, data ownership, destructive actions, webhooks, state transitions, or customer-visible records, explicitly consider:

- Auth and authorization
- RLS and database constraints
- Ownership and org/customer boundaries
- Idempotency, retries, and replayed events
- Transactions and race conditions
- Stale version handling
- Invalid state transitions
- Audit logging and manual review paths
- Database and external service failures
- Production monitors or alerts

At minimum, consider tests for happy path, wrong user, wrong org/customer, stale version, expired entity, duplicate request, replayed webhook/event, invalid amount/status/input, concurrent request, database failure, external service failure, and missing audit event.

For critical domain logic, prefer unit tests plus integration tests against the real boundary that enforces the invariant. Add RLS allow/deny tests, concurrency tests, property-style tests, or mutation testing when the risk and repo tooling justify them.

### Visible vs Invisible Errors

Not every error throws. A page can return HTTP 200, log nothing to the console, and still display `TypeError: fetch failed`, `Application error: a client-side exception`, or a leaked service-layer error message inside the rendered DOM. Standard pageerror and console.error listeners miss this entirely.

When the slice's invariant includes "users do not see error strings," the test must read the rendered page text and assert no forbidden substrings appear (e.g., `TypeError:`, `Application error:`, `Rendered more hooks`, `Something went wrong`, `fetch failed`). Asserting status < 500 and no thrown exceptions is necessary but not sufficient. Service-layer errors caught and rendered via `{result.error}` patterns are the most common source of leaked error text — handle them with friendly recovery copy rather than raw `error.message`.

### False-Pass on Generated Types

`tsc --noEmit` clean against generated type files (Next.js `routes.d.ts`, GraphQL codegen output, Supabase generated types, OpenAPI clients) is not a real green. Generated files can be stale, missing, or cached from an earlier build. If the slice's verification depends on those types, regenerate them first (run the build, run the codegen, run the dev server long enough to emit them) and re-run tsc against the fresh state. The slice is only safe if the failure mode reproduces against fresh state — and the fix holds against it.

## Runtime Verification Gate

Decide whether automated tests are enough. If the slice changes or depends on user-visible UI, browser behavior, desktop/system behavior, auth/session flows, file pickers/downloads/uploads, keyboard or pointer interactions, responsive layout, live navigation, external callbacks, or a workflow that could pass tests while failing in the actual app, run a real runtime verification.

Use the most direct tool for the surface:

- Use browser automation for web-only local or deployed app flows.
- Use Computer Use for desktop apps, OS dialogs, native windows, cross-app flows, permission prompts, file dialogs, or anything that must be observed and operated like a user on the machine.

The verification must exercise the acceptance path and at least one relevant unsafe or error path when practical. Capture evidence in the final response: tool used, route/window opened, action performed, observed result, and any screenshot/log/artifact if available.

If runtime verification is required but blocked by missing credentials, missing services, unavailable tools, broken setup, or unsafe live-side effects, do not mark `PASS`. Use `PASS WITH RISKS` only for non-critical gaps, `BLOCKED` for missing prerequisites or unsafe decisions, and `FAIL` for confirmed broken behavior.

## Final Response

When finished, respond in this format:

```text
Summary:
[what changed or what was reviewed]

Invariant:
[the core rule this preserves]

Unsafe outcomes checked:
[dangerous cases considered]

Intentional behaviour changes:
[behaviours changed on purpose by this request]

Previous intended behaviours preserved:
[older goals or guarantees checked, with evidence]

Tests:
[what tests exist or were added]

Runtime verification:
[browser/computer-use verification performed, or why it was not required/was blocked]

Subagents:
[subagents used and evidence received, or why delegation was not needed]

Issues found:
[bugs, risks, missing tests, contract drift]

Recommended fixes:
[smallest safe fixes, in priority order]

Files changed or reviewed:
[list]

Status:
PASS / PASS WITH RISKS / BLOCKED / FAIL

Confidence:
[0-100]% absolute confidence in the current slice state, with one sentence explaining the evidence and the biggest remaining uncertainty. Do not give 90%+ unless the relevant automated checks, required runtime verification, and integration/migration gates are complete.

Behaviour preservation confidence:
[0-100]% confidence that previous intended behaviours still hold except for explicitly requested changes. Explain the regression evidence, intentional changes, and any unknown prior intent.

Commit/push recommendation:
[COMMIT + PUSH / COMMIT ONLY / DO NOT COMMIT / DO NOT PUSH YET], with the reason. Explicitly say whether the current state is safe to commit, safe to push, or should be held for another slice/check/fix.

Next action:
[the single best next action: commit/push, run a missing check, apply a migration, execute the next slice from the plan, ask a product decision, or stop]

Optional next step:
[only include when there is a logical non-required follow-up; ask whether the user wants it done]
```

Use `PASS` only if the slice is safe, tested, and integration-safe. Use `PASS WITH RISKS` for acceptable non-critical gaps. Use `BLOCKED` when a product decision, missing dependency, or contract ambiguity prevents safe completion. Use `FAIL` for a concrete unsafe path, broken invariant, weak auth, broken state transition, missing critical test, or integration break.

The confidence score is not a vibe. Base it on observed evidence:

- 95-100: all relevant tests, runtime verification, migrations/integration gates, and final workspace checks are complete; only normal residual risk remains.
- 85-94: strong automated proof and no known blockers, but one low-risk verification layer is missing or not applicable with a clear reason.
- 70-84: likely correct but missing meaningful runtime/integration coverage or carrying known non-critical risks.
- 50-69: partial implementation or review; important proof is missing.
- 0-49: blocked, failing, unsafe, or mostly unverified.

The behaviour preservation confidence score is separate from implementation confidence:

- 95-100: requested changes are clearly isolated, previous intended behaviours are documented or inferred from strong repo evidence, and regression checks/runtime proof cover them.
- 85-94: strong regression evidence exists, but one low-risk prior behaviour is inferred rather than directly specified.
- 70-84: likely preserved, but regression coverage is partial or previous intent is only moderately evidenced.
- 50-69: the new path works, but older intended behaviour is weakly understood or lightly tested.
- 0-49: known regression, unclear previous intent on a risky surface, or no meaningful regression proof.

Do not recommend `COMMIT + PUSH` unless the slice is at least `PASS` or a clearly explained `PASS WITH RISKS`, the intended files are known, and no required verification is still pending. Recommend `COMMIT ONLY` when local checkpointing is useful but pushing/deploying should wait for another slice, migration, or external decision. Recommend `DO NOT COMMIT` for failed, unsafe, or messy partial states.

Only include `Optional next step` when it is genuinely useful and not required by the current slice. Phrase it as a direct question before starting that work, for example: `Optional next step: Should I add coverage for the adjacent retry path next?`
